# Skill: bastion/life-log

## Objective

Maintain a semantic history of interactions per persona, allowing Bastion to:

1. **Remember** past conversations without user needing to repeat context
2. **Search** most relevant records to enrich responses (RAG)
3. **Analyze** usage patterns per persona over time
4. **Detect** inactive personas for proactive suggestions

---

## CLI Commands

> IMPORTANT: CLI Command
> As you are an OpenClaw agent, you must invoke all operations via command line (`exec python3 ...`). Do not attempt to interpret Python code natively.



```python
from skills.life_log.factory import Settings, create_adapter
from datetime import datetime, timezone

# Instantiate via factory (reads DB_STRATEGY from environment)
settings = Settings.from_env()
adapter = create_adapter(settings)

# Log an interaction
interaction_id = await adapter.log_interaction(
    persona="tech-lead",
    intent="code-review",
    tools=["github-integration", "code-review-helper"],
    embedding=[0.1, 0.2, ...],   # generated by the configured LLM
    timestamp=datetime.now(tz=timezone.utc),
)

# Semantic search (RAG)
similar = await adapter.search_similar(
    query_embedding=[0.1, 0.2, ...],
    persona="tech-lead",          # None to search across all personas
    limit=5,
    threshold=0.65,
)

# Summary of last 7 days
summary = await adapter.get_persona_summary(
    persona="tech-lead",
    days=7,
)
```

## Required Fields for Each Record

| Field | Type | Description |
|---|---|---|
| `id` | `str` (UUID) | Unique identifier generated automatically |
| `persona` | `str` | Slug of active persona in interaction |
| `intent` | `str` | Executed intent (e.g., "code-review", "deploy-check") |
| `tools` | `list[str]` | Names of tools used in interaction |
| `embedding` | `list[float]` | Input embedding vector (generated by LLM) |
| `timestamp` | `datetime` | UTC date/time of interaction |

---

## Semantic Search

### Threshold and Limit

- **default threshold:** `0.65` — only results with cosine similarity ≥ 0.65
- **default limit:** `5` — maximum 5 results per search
- Results ordered by descending similarity

### Embedding Generation

Embeddings are generated by the LLM configured in OpenClaw (Gemini, OpenAI, Groq).
The skill doesn't generate embeddings directly — receives the vector already calculated by orchestrator.

### Context Injection in Prompt

The orchestrator uses `search_similar` before assembling each response:

```
1. Generate embedding from user input
2. Call search_similar(query_embedding, persona=active_persona, limit=5, threshold=0.65)
3. Inject returned records as historical context into the system prompt
4. Respond with enriched context
```

## DB_STRATEGY Pattern

The skill uses the **Protocol/Adapter** (hexagonal) pattern to decouple business logic from the persistence backend.

### Switching from SQLite to Supabase

Just change one variable in `.env`:

```env
# Default — local SQLite, zero external dependency
DB_STRATEGY=sqlite
SQLITE_PATH=db/life-log.db

# Optional — Supabase (PostgreSQL + pgvector)
DB_STRATEGY=supabase
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_KEY=your-anon-or-service-key
```

No skill code needs to change. The factory (`create_adapter`) instantiates the correct adapter based on `DB_STRATEGY`.

### Available Adapters

| DB_STRATEGY | Adapter | Dependency |
|---|---|---|
| `sqlite` (default) | `SQLiteLifeLogAdapter` | None (stdlib) |
| `supabase` | `SupabaseLifeLogAdapter` | `supabase-py` + Supabase project |

---

## SQLite Persistence

### Database Location

```
db/life-log.db   ← created automatically on first write
```

The `db/` directory is created automatically if it doesn't exist.

### Schema

```sql
CREATE TABLE interactions (
    id        TEXT PRIMARY KEY,   -- UUID v4
    persona   TEXT NOT NULL,      -- persona slug
    intent    TEXT NOT NULL,      -- executed intent
    tools     TEXT NOT NULL,      -- JSON array of tool names
    embedding BLOB NOT NULL,      -- float32 little-endian
    timestamp TEXT NOT NULL       -- ISO 8601 UTC
);

CREATE INDEX idx_interactions_persona   ON interactions (persona);
CREATE INDEX idx_interactions_timestamp ON interactions (timestamp);
```

### sqlite-vec

When the `sqlite-vec` extension is installed, it is loaded automatically for native vector search. If not available, the adapter falls back to pure-Python cosine similarity — works offline with no native dependencies.

## Architecture (Hexagonal)

```
LifeLogProtocol (port)
    ├── log_interaction(persona, intent, tools, embedding, timestamp) → str
    ├── search_similar(query_embedding, persona, limit, threshold) → list[InteractionRecord]
    └── get_persona_summary(persona, days) → list[InteractionRecord]

SQLiteLifeLogAdapter (default adapter)
    ├── Creates db/life-log.db automatically
    ├── Uses sqlite-vec when available
    └── Falls back to pure-Python cosine similarity

SupabaseLifeLogAdapter (optional adapter)
    ├── Uses supabase-py + pgvector
    └── Stub — requires full implementation for production

Settings + create_adapter() (factory)
    └── Instantiates the correct adapter based on DB_STRATEGY
```

## When This Skill is Triggered

The orchestrator calls `log_interaction` at the end of each relevant interaction:


The orchestrator enriches context before responding:

## Edge Cases

### Database doesn't exist on first read

The `SQLiteLifeLogAdapter` creates the file and schema automatically before any operation. No manual initialization required.

### Embedding with different dimension

`search_similar` compares embeddings using cosine similarity. If vectors have different dimensions, a `ValueError` is raised. Always use the same embedding model.

### Threshold too high (no results)

If no record reaches the threshold, `search_similar` returns an empty list. The orchestrator must handle this case and respond without historical context.

### Persona with no history

`get_persona_summary` returns an empty list for personas with no interactions in the period. This is not an error — it indicates a new or inactive persona.
